# artefacts — project guide

A self-hosted, multi-tenant **"Claude Code in the browser"**: a chat-driven agent builds a web app across a persistent virtual filesystem, with a live in-browser preview. Deploys to **Coolify**.

## Layout
- Root: `docker-compose.yml` (db + one-shot `migrate` + `web` + `landing` + `admin`), `.env.example`.
- `web/`: the Next.js 16 app (App Router, TypeScript, Tailwind v4, Drizzle, Auth.js v5). All builder work happens here.
- `landing/`: separate, anonymous, stateless marketing page (own Dockerfile + own domain, **served from the compose stack** — not a standalone Coolify resource; see "Deployment notes").
- `admin/`: separate **read-only** admin panel (own Dockerfile + own domain, **served from the compose stack** — not a standalone Coolify resource) — see "Admin panel" below.

## Commands (run inside `web/`)
- `npm run dev` — local dev. `npm run build` — production build (also typechecks). `npm run lint`.
- `npm run db:push` — apply the Drizzle schema to the DB (used by the `migrate` compose service). `db:generate` for versioned SQL.
- `npm run spike` — boots a throwaway real Postgres (`embedded-postgres`, devDep) and runs `scripts/security-spike.mjs`, the per-project DB **isolation** regression test. Keep it green.
- `npm run e2e:publish-db` — boots a throwaway real Postgres, pushes the schema, provisions + **publishes** a DB-backed project, then drives the **real route handlers** (`app/api/appauth` + `app/api/appdb` + `app/serve`) as the published `<slug>.apps.<domain>` host: signup/login/me, per-user CRUD + owner_id RLS isolation, host gating (404/403), and `window.artefacts` SDK injection. The end-to-end regression test for "a published app with a database" (`scripts/publish-db-e2e.mjs`; imports the real handlers under `scripts/e2e-hooks.mjs`, a Node module hook that resolves the `@/` alias + shims `server-only` — no Next server, no bundler). Keep it green.
- After meaningful changes, run `npm run build` **and** `npm run lint` — both must be clean.

## Architecture & invariants (do not break these)
- **Client-side execution only.** The agent never runs shell commands or touches the host disk. Its tools operate on a **Postgres-backed virtual filesystem** scoped by `userId`/`projectId` (`lib/projects.ts`). This data-layer scoping is the multi-tenant isolation boundary — keep every file/project query scoped.
- **Agent = Claude Agent SDK** (`@anthropic-ai/claude-agent-sdk`): `query()` in `lib/agent/run.ts`, custom VFS tools via `createSdkMcpServer`/`tool()` in `lib/agent/tools.ts`. The agent is restricted to those tools via **`tools: []`** — this strips ALL built-in Claude Code tools (Bash/Write/Read/WebFetch/AskUserQuestion/…) while leaving the MCP servers untouched. **`allowedTools` alone does NOT restrict availability** (it is only a permission auto-approve list), and `permissionMode: 'bypassPermissions'` approves everything that exists — without `tools: []` the builder had a live Bash in the web container and called the UI-less interactive `AskUserQuestion` (user saw no question). Clarifying questions are plain chat text (enforced in the system prompt). Also `settingSources: []` (hermetic). Default build model `claude-opus4-8`, set via `CORTECS_BUILD_MODEL` (or live in the admin `app_setting`). **NOTE:** model ids are **cortecs.ai catalog ids**, which differ from Anthropic's canonical ids — Opus 4.8 on cortecs is `claude-opus4-8` (no hyphen after "opus"), NOT `claude-opus-4-8`. Verify names at `GET https://api.cortecs.ai/v1/models`. **cortecs is an LLM gateway with custom ids, so Claude Code would reject them** ("model may not exist / no access") — `runAgent` therefore passes gateway env to the spawned CLI: `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1` + `ANTHROPIC_CUSTOM_MODEL_OPTION` (whitelist the build id) + `ANTHROPIC_DEFAULT_{OPUS,SONNET,HAIKU}_MODEL` (tier aliases + background calls), from `cortecsTierModels()`. Don't remove these — the builder can't reach cortecs models without them.
- **Design media (icons + fonts + stock photos):** `lib/agent/tools-media.ts` (`media` MCP server) gives the builder agent (a) two **fixed offline icon libraries** — Lucide UI icons (`lucide-static`) + Simple Icons brand logos (addressed `brand:<slug>`) — resolved by `lib/agent/icons.ts` to inline-ready `currentColor` SVG (`search_icons`/`get_icons`), (b) a **bundled offline webfont catalog** (`search_fonts`/`add_font` — see "Design DNA" below), and (c) **Pexels stock photos** (`lib/agent/pexels.ts`): `search_stock_photos` returns candidates **with visible thumbnail previews** (vision), `add_stock_photo` downloads a sized variant (640/1280/1920 px, ≤8 MB) server-side and writes it into the VFS via `writeBinaryFile` + `asset_changed` — so the generated app still makes **no external requests**. Photos need env **`PEXELS_API_KEY`** (secret → env-only, optional): without it the photo tools return an explanatory error and the system prompt says photos are unavailable — the prompt is built per run via `buildSystemPrompt({stockPhotos})` (`lib/agent/system-prompt.ts`). The prompt also directs the agent to make the FIRST version visually complete (library icons, real fonts, fitting imagery) and to add CSS animations/decorative SVG **only when they fit the theme** (with `prefers-reduced-motion` fallback).
- **Design DNA (anti-"generic AI look"):** every project gets a deliberate visual identity instead of the SaaS-canon default (Inter, purple gradient hero, 12–24px cards). Three layers: **(1)** `lib/design-worlds.ts` — ~15 curated "style worlds" as **constraint data, never markup** (epoch, inspiration brands, allowed font pairings by catalog id, grid/shape/motion rules, color philosophy, world-specific forbidden list, mutation ranges; includes one "Modern Product" world so the canon stays a conscious choice). **(2)** The route **samples** 6 candidate worlds + per-project mutations (`sampleWorldCandidates` — server entropy, never LLM choice); the interview LLM picks the 3 best fits and instantiates them (project name/vibe/5-hex palette; fonts zod-validated to be exactly one offered pairing — schema built per call in `lib/agent/interview.ts`, mutations merged server-side so the LLM can't corrupt the dice). **(3)** The user picks one via visual **style tiles** in the interview card (real font specimens via public `/api/fonts/[...slug]`, immutable-cached). The chosen style is composed into **`/DESIGN.md`** by `lib/agent/design.ts` (**deterministic template, no LLM call**) on all three paths — answered, skipped (system picks one of the 3 generated styles), interview-generation-failure (fallback world) — and injected every turn as a binding "Design DNA" prompt section. **The prompt is assembled INSIDE the stream, after interview handling**, so a freshly written DNA is always included. `/DESIGN.md` is internal exactly like `/CONCEPT.md` — the single predicate `isInternalVfsPath` (`lib/concept.ts`) keeps them out of export/serve/publish-signature (and the live SSE `file_changed` filter). They ARE surfaced **read-only** in the workspace code tree so the user can read the concept/design: `getClientFiles` returns them on a separate `internal` channel (never in `files`/`assets`, so they don't affect the preview's single-vs-multi-file detection or the publish signature), which `sandpack-workspace.tsx` merges into the read-only editor only. The system prompt additionally carries a **global anti-canon list** (named forbidden defaults) and instructs the agent to write its own `/DESIGN.md` when none exists (older projects).
- **Webfonts (bundled, offline):** `lib/agent/fonts.ts` + `lib/agent/font-catalog.json` — 29 OFL families as `@fontsource/*` deps (deliberately **no Inter/Roboto/Open Sans**), spanning serif/sans/display/mono/slab/script. `add_font` fs-reads woff2 cuts from node_modules and writes them to `/assets/fonts/` (`writeBinaryFile` + `asset_changed`, ~13 KB/cut latin, ≤6 files/family) and returns ready `@font-face` CSS; both preview modes already handle woff2 (serve route content-type + srcDoc CSS-`url()` data-URI inlining). Because the files are **fs-read at runtime**, `next.config.ts` `outputFileTracingIncludes` MUST keep `./node_modules/@fontsource/**/*` (under `/api/agent` AND `/api/fonts/**`) — without it only the Docker standalone breaks. **`npm run check:fonts`** verifies every declared catalog cut exists on disk and every style-world pairing id resolves — keep it green.
- **Output contract:** `/index.html` is always the entry point with the agent's own CSS/JS inlined and no external network requests. The app is a **real multi-file VFS**, though: uploaded files the user wants embedded become **binary asset files** (e.g. `/assets/logo.png`, `file` rows with `encoding='base64'`) referenced by relative path. So a project can be multiple files. Download is **adaptive** — a lone `/index.html` downloads as `.html`, otherwise a **ZIP** of the whole VFS (`app/api/projects/export`, `fflate`). The subdomain serve route serves every VFS file (not just index.html). (Uploads that are NOT embedded stay reference-only in the `attachment` table / "Dateien" tab; `embed_attachment` *moves* an upload into the VFS.)
- **Binary in the VFS:** `lib/projects.ts` `writeBinaryFile`/`readFileRaw`/`getClientFiles`; the client never receives base64 — binary files reach it as metadata (`assets: {mimeType,size,hash}`) and render as placeholders in the Sandpack tree. Version snapshots store binary entries as `{content,encoding,mimeType}` (text stays a plain string; both forms are read back). `canonicalSignatureMap` (`lib/files-signature.ts`) keeps publish-dirty detection consistent across client/server by hashing binary content.
- **Preview** of `/index.html` (`components/sandpack-workspace.tsx`), NOT Sandpack's preview (its runtime is CodeSandbox-hosted and fails self-hosted). Sandpack is used only for the offline file tree + read-only code viewer. Two modes:
  - **Subdomain serving** (when `APPS_DOMAIN` is set): an `<iframe src>` to `preview-<projectId>.apps.<APPS_DOMAIN>`, served by `app/serve/route.ts` and gated by a signed token (`lib/preview-token.ts`). On its own origin the iframe **uses `allow-same-origin`** (safe — a different origin than the builder; required for real cookies/storage/auth later). See "Way 3" below.
  - **Fallback `<iframe srcDoc>`** (when `APPS_DOMAIN` is unset): inline, sandboxed **without `allow-same-origin`**. For a multi-file project the srcDoc is fetched from `app/api/projects/render` which **inlines VFS assets as data URIs** (relative paths don't resolve in a srcDoc iframe); a lone `/index.html` is shown directly. The multi-file serve path needs `proxy.ts` to forward the requested path via **`x-app-path`** (it pins `pathname='/serve'`).
- **Agent route** (`app/api/agent/route.ts`): authenticated SSE. Streams `assistant_text`, `tool_use`, `file_changed` (text), `asset_changed` (binary), `file_deleted`, `attachments_changed`, and a final `files` snapshot (`{files, assets}`); `runtime = 'nodejs'`. The client (`components/workspace.tsx`) parses the stream and updates chat + preview live.
- **Concept interview (first prompt only):** the very first user prompt of a project does NOT build. Instead the route runs one small LLM call (`lib/agent/interview.ts`, task `interview` → `CORTECS_INTERVIEW_MODEL`, OpenAI path via `cortecsChat`, maxTokens ~2600) that generates 3 single-select direction questions + **3 instantiated style directions from the server-sampled world candidates** (see "Design DNA"), persisted as a `message` row with `kind='interview'` (content = JSON `InterviewState` **v2**; `lib/interview.ts` is **versioned** — legacy v1 palette rows keep parsing/rendering forever) and streamed as an `interview` SSE event. The client renders an interactive card (`components/interview-card.tsx`); clicking a style tile (after answering all 3 questions) POSTs `{interview:{messageId,selections,styleId}}` (v1 rows: `paletteId`) back to the agent route, which marks the row answered, **writes the chosen style's `/DESIGN.md`**, and runs the build with the answers as binding direction (content decisions → `/CONCEPT.md`). Skip link + typing-past-the-card both mark it `skipped` — on skip the server picks one of the 3 generated styles as system-chosen DNA; generation failure falls back to a normal build with a fallback-world DNA. Answered interviews replay into later turns' transcripts as a readable Q→A block (never raw JSON).
- **Live progress:** `runAgent` sets `includePartialMessages: true`; the route turns the raw `stream_event`s into token-level `assistant_text` deltas plus `tool_start`/`tool_progress` (tool-input generation with sniffed `path` + char count — the minutes-long silent stretch of a big `write_file`). Committed blocks are deduped server-side (`streamedTextChars`), so text is emitted exactly once and everything degrades to block-at-once if deltas don't arrive; persistence still reads ONLY complete assistant messages. Client: pending (pulsing) tool rows finalize FIFO on the committed `tool_use`; a 15 s SSE heartbeat comment keeps proxies from cutting idle streams. Verified by `npm run canary:partial` (`scripts/partial-stream-canary.mjs`) — run it WITH `CORTECS_API_KEY` (e.g. on the server) to prove the gateway forwards deltas promptly; `--direct` only proves SDK mechanics.

## Way 3 — real per-project backend (in progress)
Direction (decided): give generated apps a **real database + real end-user auth** so they stop faking logins/data, and can be published. Chosen approach: our **own data+auth API on the existing Postgres**, **one schema per project**, **per-project subdomains**. This will relax the "no backend" stance above for *generated apps* — the builder/agent invariants stay.
- **Phase 1 — subdomain serving (built, deployed):** `proxy.ts` routes `*.apps.<APPS_DOMAIN>` requests to `app/serve/route.ts` (serves the project's VFS `/index.html`, injects `window.__ARTEFACTS__`), gated by `lib/preview-token.ts`; host parsing in `lib/app-host.ts`. The builder stays on the main domain. Needs env `APPS_DOMAIN` + wildcard DNS/TLS (see Deployment). The proxy rewrites `/`→`/serve` and **must forward the real host via `x-app-host`** — `NextResponse.rewrite` loses it (see Next.js gotchas).
- **Isolation model (proven, not yet wired):** `lib/appdb/sql.ts` (pure, injection-safe SQL/identifier + constrained-query builders) and `lib/appdb/exec.ts` (runs every tenant query under `SET LOCAL ROLE proj_<id>_role` + pinned `search_path` + `app.end_user_id` GUC). Cross-tenant isolation = a per-project low-privilege Postgres role (USAGE/CREATE only on its own schema); end-user isolation = Postgres **RLS**. Enforced in Postgres, never by trusting the generated client. Verified by `npm run spike`.
- **Phase 2/3 — data + end-user auth (built):** opt-in per-project database, surfaced to the agent and to generated apps.
  - **Schema as a file:** the agent writes `/database.sql` (plain `CREATE TABLE`s, additive/idempotent) and calls the **`apply_schema`** agent tool (`lib/agent/tools-db.ts`), which `ensureProvisioned`s the schema+role on first use and runs the DDL **under the project role** (`lib/appdb/provision.ts` → `applyTenantDdl`). Project flags: `databaseEnabled` + `dbProvisionedAt`. The system prompt makes the agent **ask before provisioning** (a DB means the export now carries a DB to host).
  - **One convention drives privacy:** any table with an `owner_id uuid` column is auto-hardened by `applyOwnerSecurity` — its default becomes the `app.end_user_id` GUC (auto-stamped on insert) and per-owner **RLS** is enabled. Tables without `owner_id` are shared app data.
  - **Same-origin runtime API:** `app/api/appdb` (constrained CRUD → `buildSelect/…` → `runTenantQuery`) and `app/api/appauth` (signup/login/logout/me against the **`app_user`** table; per-project HMAC session cookie via `lib/appdb/app-auth.ts`). `/api/*` is excluded from the proxy matcher, so these reach the app subdomain directly with the real Host; `lib/appdb/app-request.ts` resolves projectId + gates preview hosts by the `pt` token. The injected **`window.artefacts.db` / `.auth`** SDK (in `app/serve/route.ts`, only when `databaseEnabled`) is the app's only DB interface.
  - **Export:** when `databaseEnabled`, the ZIP adds `database-data.sql` (INSERTs from `dumpTenantData`, run with `row_security=off`) + `DATABASE.md`; the schema ships as the normal `database.sql` VFS file. Regression-tested by `npm run spike` (cross-tenant + RLS + owner_id auto-stamp).
- **Data viewer + row editor (built):** the workspace "Daten" tab (`components/data-view.tsx` + ownership-gated `app/api/projects/db`) lists the project's tables (🔒 = `owner_id` per-user) and pages through rows; `listTenantTableMeta`/`readTablePage` in `provision.ts` read with `row_security=off` so the owner sees all end-users' rows. The tab shows only when `databaseEnabled` (passed as `initialDatabaseEnabled`, flipped on the `database_changed` SSE event). **Row editing + delete (built 2026-07-06):** the route also has `PATCH`/`DELETE`; `updateTableRow`/`deleteTableRow` in `provision.ts` run as **admin with `row_security=off`** (so the owner can edit/delete ANY end-user's row) and target exactly one row via its **primary key** — `pkFilters` requires the full PK, so a write can never run without a WHERE or match >1 row; a table with **no primary key stays read-only** (`readTablePage` returns `primaryKey: []`, the UI hides the actions). The edit modal keeps PK columns read-only + a per-field NULL toggle. Owner edit/delete round-trip is asserted by `npm run e2e:publish-db`.
- **Full-app backup + daily snapshots (built):** one `project_backup` table + `lib/backup.ts` **replace** the file-only `artifact_version` as the single snapshot/restore mechanism. A backup is a self-contained JSON blob: files (same shape as the old snapshot, so publish stays byte-compatible) + per-project DB `{ddl, dataSql}` (`dataSql = serializeTenantDump`) + `app_user` rows (incl. hashes) + `attachment` rows (incl. base64) + project settings. `createBackup(projectId, kind)` runs per file-changing agent turn (`auto`), on publish (`publish`), and daily for published apps (`daily`) (`manual` reserved). **`restoreBackup`** is the hard part: DB reset = `dropTenantSchema` (admin — the schema is admin-owned) → `ensureProvisioned` → `applyTenantDdl` (backup's pinned DDL) → `restoreTenantData` (admin, `row_security=off` + `session_replication_role=replica` + **explicit `owner_id`** so ownership is preserved) → `applyOwnerSecurity`; then files, `app_user`+`attachment` (delete+reinsert, original ids/createdAt kept), settings (slug guarded by `isSlugFree`). A published project keeps serving its **frozen** backup throughout restore — publish now reads `publishedBackupId` (legacy `publishedVersionId` fallback via `loadPublishedFilesMap`) — so the public app never shows a half-restored state; restore does NOT republish. **Daily** = token-protected `POST /api/cron/backup` (bearer `BACKUP_CRON_SECRET`, `lib/cron-secret.ts`) hit by a **Coolify Scheduled Task** (`0 3 * * *`), idempotent per UTC day, only `published=true`. **Retention** (`pruneBackups`, on every create + in cron): keep < `BACKUP_RETENTION_DAYS` (7), cap `auto` at 20, ALWAYS keep the published + newest. Backup→restore roundtrip (RLS + `owner_id` auto-stamp survive a full DB restore) is asserted by `npm run spike`. `BACKUP_ENABLED`/`BACKUP_RETENTION_DAYS` are admin-editable; `BACKUP_CRON_SECRET` is env-only (unset → cron route 403s). **Known gaps:** SERIAL/IDENTITY sequences aren't reset to `max(id)` on restore (generated apps use uuid PKs in practice); `artifact_version` + `projects.published_version_id` are still present for the transition (drop in a later PR after backfill).
- **Publish-with-DB is VERIFIED end-to-end** (2026-07-06, `npm run e2e:publish-db`): a published `<slug>.apps.<domain>` app resolves **ungated**, injects the `window.artefacts` SDK, and its end-users can **sign up / log in** and run **per-user CRUD with owner_id RLS** — proven against the real route handlers. Key insight the flow relies on: on publish only the **FILES** are frozen (into the publish backup); the **DB is served LIVE** (the data API runs `runTenantQuery` against the live tenant schema by `projectId`), so end-user writes after publish land in the live schema and are immediately visible. So the old "Phase 4 = make DB publish work" is effectively **done for the data path**.
- **Next (remaining Phase 4 — STILL OPEN):**
  1. **App-internal admin role** — an *generated* app cannot build an owner-facing admin dashboard over other users' private `owner_id` data, because RLS confines every runtime query to the caller's own rows; the only cross-owner view is the builder's "Daten" tab (owner-only, not part of the shipped app). No RLS-bypass "admin" role for end-users yet. **OPEN.**
  3. **End-user password reset / email verification** — not built (`app_user` has no reset flow). **OPEN.**
  4. **Destructive migrations** — rename/drop columns are not auto-applied (only additive/idempotent DDL). **OPEN.**
  - (Point 2, **row editing/delete in the "Daten" viewer**, is now **DONE** — see the data-viewer bullet above.)

## Next.js 16 gotchas (see `web/AGENTS.md` — read `node_modules/next/dist/docs/` before writing Next code)
- Middleware is renamed to **`proxy.ts`** (not `middleware.ts`).
- Route handler `params` are async; route handlers use Web `Request`/`Response`.
- Auth.js v5 split config: edge-safe `auth.config.ts` (used by `proxy.ts`), full `auth.ts` (DB + bcrypt). **JWT sessions** (required by the Credentials provider).
- **`NextResponse.rewrite` drops the original `Host`.** It re-derives the request host from `req.nextUrl` (the internal/connection host, e.g. `127.0.0.1:3000`), so a route handler that reads `request.headers.get("host")` after a rewrite sees the *wrong* host. `proxy.ts` therefore pins the real host on an **`x-app-host`** header (`NextResponse.rewrite(url, { request: { headers } })`) and `app/serve/route.ts` reads `x-app-host` first, falling back to `host` for direct hits. This was the subdomain-preview `404 Not found` bug: the route worked when hit directly (`/serve`) but 404'd when reached via the `/`→`/serve` rewrite. Symptom of the bug: a valid `preview-<uuid>` host returns `404 "Not found"` instead of `403 "Forbidden"`.

## Deployment notes
- `web/Dockerfile` is **Debian/glibc (`node:24-slim`)**, not Alpine: the Agent SDK ships a native glibc `linux-x64` CLI binary.
- `next.config.ts` uses `outputFileTracingIncludes` to force `@anthropic-ai/claude-agent-sdk` **and its sibling native packages** (`@anthropic-ai/claude-agent-sdk-*`) into the standalone output — tracing misses them otherwise.
- Coolify: set `AUTH_SECRET`, `ANTHROPIC_API_KEY`, `POSTGRES_PASSWORD`; assign a domain and set `AUTH_URL` to the builder's public `https://…` URL. Serve over **HTTPS** (secure context needed for `crypto.randomUUID` etc.).
- **Per-project DB (Way 3 Phase 2/3):** the `DATABASE_URL` user must have **`CREATEROLE`** (provisioning runs `CREATE ROLE proj_<id>_role` + `GRANT … TO CURRENT_USER`) and be able to bypass RLS for the export dump (superuser, as in our Postgres). End-user sessions are signed with a key derived from `AUTH_SECRET` (no new secret). DB features require `APPS_DOMAIN` (same-origin SDK) — in the `srcDoc` fallback `window.artefacts` is simply not injected. The new `database_enabled`/`db_provisioned_at` columns and the `app_user` table are added by the normal `migrate` push (no populated-table unique → no truncate prompt).
- **`APPS_DOMAIN`** (Way 3): generated apps are served from `<label>.apps.<APPS_DOMAIN>`; the builder runs on a **separate** hostname (e.g. `app.<domain>`). Requires a **wildcard DNS** record (`*.apps`) and **wildcard TLS** (DNS-01 — the HTTP-01 challenge can't issue wildcards). If `APPS_DOMAIN` is unset the preview falls back to inline `srcDoc` (no real origin). Deployment-instance specifics (current domain, cert/Traefik wiring) live in agent memory, not here.
- **Every env var the container needs must be listed in the `web` service's `environment:` block in `docker-compose.yml`.** Coolify only injects a UI variable into a compose service if that service **references** it (`VAR: ${VAR}`) — a UI value alone does **not** reach the container otherwise. `APPS_DOMAIN` and `AUTH_URL` are now wired there explicitly for this reason. After changing a Coolify env var, the `web` service must be **redeployed/restarted** (the Node process reads `process.env` only at start).
- **Migrations use `drizzle-kit push --force` in the non-TTY `migrate` container — never add a column `.unique()` constraint to a table that already has rows.** `push` prompts ("…add unique constraint to a table which contains N items. Truncate?") even with `--force`, and the prompt **hangs/aborts with no TTY** → the migration fails, the column is never created, and the already-started `web` (new code) crashes with `column "x" does not exist`. Use a **`uniqueIndex(...)` in the table's index array instead of `text().unique()`** — unique *indexes* don't trigger the truncate prompt. (Same applies to any populated-table unique.) If it already bit you: add the columns by hand on the DB (`ALTER TABLE … ADD COLUMN IF NOT EXISTS …; CREATE UNIQUE INDEX IF NOT EXISTS …`) — the running container picks them up with no redeploy — then land the `uniqueIndex` schema so DB and schema match and the next `push` is a no-op.
- **VPS disk full → Coolify Redis `MISCONF` → every deploy breaks (looks like "Coolify hangs / won't run").** Each multi-stage Next build leaves GBs of Docker **build cache**; when `/` fills up, Coolify's internal `coolify-redis` can't write its RDB snapshot → `stop-writes-on-bgsave-error` blocks its job queue. Fix: `docker builder prune -af && docker image prune -af` (freed ~25 GB on our box), then unblock Redis: `docker exec coolify-redis sh -c 'redis-cli -a "$REDIS_PASSWORD" --no-auth-warning BGSAVE'`. **Prevent:** enable Coolify **Settings → Advanced → Docker Cleanup** (~70 %) and give the VPS **swap** (small-RAM box with 0 swap OOM-kills builds → redeploy loop): `fallocate -l 4G /swapfile && chmod 600 /swapfile && mkswap /swapfile && swapon /swapfile && echo '/swapfile none swap sw 0 0' >> /etc/fstab`.
- **One app = ONE Coolify deployment. Do not deploy `landing`/`admin` both as a compose service AND a standalone Coolify resource.** They live in `docker-compose.yml` for local `docker compose up` and are meant to be served from the deployed compose stack (resource `kubikraum-apps`). A second standalone copy can grab the domain and route to a **stale build with empty/different env** — this caused a multi-hour "login rejects the correct password" hunt: `admin.kubikraum.digital` pointed at an old standalone admin (old code, empty `ADMIN_*`) while every env fix + redeploy hit the **idle compose copy**. Set env on the resource that actually serves the host, and **redeploy** (env is read only at process start).
- **Debugging "which container actually serves this host":** TLS terminates at Traefik, so Traefik→container is **plaintext** — `tcpdump -i any -A 'tcp port 3000 or tcp port 3002'` during a request shows the real body + the dst IP; map that IP to a container via `docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}} {{end}}{{index .Config.Labels "coolify.resourceName"}}'` and the `Host(...)` in `.Config.Labels`. **Never trust `docker ps | grep <name> | head -1`** — it silently inspects the wrong instance when a duplicate/stale container exists (the core reason the admin bug took so long).

## Admin panel (`admin/`)
A separate Next.js 16 app (own Dockerfile, served **from the compose stack** on its own subdomain, e.g. `admin.kubikraum.digital` — **no** `basePath`; NOT a standalone Coolify resource, see the duplicate-deploy warning under "Deployment notes"). Mostly **read-only** (lists users + apps with costs; editing user data is a later phase), but it **writes two config tables** the builder reads: `mail_template` (`/mail`) and `app_setting` (`/settings`). Mirrors `landing/`'s build setup (`node:24-slim`, `output: 'standalone'`).
- **Auth — ENV-based, no Auth.js.** Single login checked constant-time against `ADMIN_USER`/`ADMIN_PASSWORD`; session is an **HMAC-signed cookie** (`admin_session`) using `ADMIN_SECRET`. All crypto is **Web Crypto** in `admin/lib/auth.ts` so the same helpers run in both `admin/proxy.ts` (gates everything except `/login`) and the login/logout server actions. Cookie is `httpOnly` + `secure` in prod.
- **Data — reads the builder's SAME Postgres** via `DATABASE_URL`. `admin/lib/schema.ts` is a **read-only subset** of `web/lib/db/schema.ts` (`user`, `project`, `user_credit`, `usage_event`) — its **column/table names must stay in sync** with the builder schema. Queries in `admin/lib/queries.ts`: per-user *consumed* = `SUM(usage_event.billed_eur)`, *available* = `user_credit.balance_eur`.
- **Pages:** `/` overview totals, `/users` (app count + consumed + balance), `/apps` (project list + owner), `/coupons` (list codes + redemptions, create admin codes — see "Coupons"), `/mail` (editable mail templates), `/settings` (editable operational settings). Server components, `export const dynamic = 'force-dynamic'`.
- **Editable operational settings (`/settings` → `app_setting`).** Non-secret Cortecs (models, base URLs, price-TTL), Billing (margin, free grant, fee multiplier, cache price ratios — cache read 0.1×/write 1.25× of the input price while the Cortecs catalog exposes no cache prices; billing cache tokens at the full input rate overcharged cache-heavy agent turns ~3×) and SMTP (host/port/user/secure/from) values. The builder reads them via `web/lib/settings.ts` with precedence **DB value > `process.env[key]` > code default** (`key` == the env var name), cached in-process ~30 s — so changing e.g. `CORTECS_BUILD_MODEL` in the admin takes effect **without a redeploy**. Secrets (`CORTECS_API_KEY`, `SMTP_PASS`) are env-only and deliberately NOT in the table/UI. The field catalog lives in `admin/app/(dashboard)/settings/fields.ts`; the config accessors in `web/lib/cortecs/config.ts` + `web/lib/mail.ts` are therefore **async**.
- **Deploy:** needs env `DATABASE_URL`, `ADMIN_USER`, `ADMIN_PASSWORD`, `ADMIN_SECRET` (all wired in the `admin` compose service + root `.env.example`). Like the builder, needs its own DNS + TLS (HTTP-01 works — it's not a wildcard). The Dockerfile builds with a placeholder `DATABASE_URL`/`ADMIN_SECRET` (the pg pool opens lazily on first query). Local: `docker compose up` exposes it on port 3002. **Serve it from the compose stack, not a second standalone Coolify resource** — see the duplicate-deploy warning under "Deployment notes" (this is what caused the long admin-login hunt). Set `ADMIN_*` on the resource that serves `admin.<domain>`.
- **Login form uses `useActionState`, whose POST sends the fields wire-prefixed** (`_1_username`/`_1_password` — React's bound-action encoding, visible in a tcpdump). `admin/app/login/actions.ts` therefore reads them via a defensive `readField()` that also matches the `_N_<name>` key, so a `formData.get("username")` that returns null can't silently make every login fail. `ADMIN_USER`/`ADMIN_PASSWORD` are read at **runtime** (not inlined at build).

## Coupons / referral credit
Referral + admin-issued coupons that grant EUR credit. Two tables in `web/lib/db/schema.ts` (mirrored read/write in `admin/lib/schema.ts`): **`coupon`** (`kind` `referral`|`admin`, `owner_id`, per-code `recipient_amount_eur`/`referrer_amount_eur`, `max_redemptions`, `expires_at`, `active`) and **`coupon_redemption`** (one row per redemption; `referrer_reward_status` `pending`|`granted`|`expired`|`none`). Both are NEW empty tables → **partial `uniqueIndex`** (not `.unique()`) is safe for `migrate` push. Column style matches the codebase: snake_case data columns + camelCase `createdAt`/`updatedAt`.
- **Logic:** `web/lib/coupons.ts` — `redeemCoupon()` (one tx, coupon row `FOR UPDATE`): for a **referral** coupon BOTH rewards are recorded **`pending`** (redeemer welcome bonus `recipient_reward_status='pending'` + referrer reward) and **nothing is paid out yet** — both settle only when the redeemer subscribes; an **admin** coupon still credits the redeemer immediately (`recipient_amount_eur` → `user_credit.balance_eur`, status `granted`). `ensurePersonalCoupon()` (one referral code per user), `getUserCouponInfo()` (also returns `hasSubscription` + the redeemer's `pendingWelcomeEur`). Rules (also DB-enforced by partial unique indexes): a code once per user, **at most one referral redemption per user** (welcome bonus), no self-redeem, respect `active`/`expires_at`/`max_redemptions`.
- **Builder UI:** account modal (`web/components/user-menu.tsx`) has a **"Gutschein" tab** — redeem a code + activate/show your own code with redemption stats. A redeem fires `CREDIT_CHANGED_EVENT` (`web/lib/credit-events.ts`) so the composer balance in `workspace.tsx` updates live. Routes: `POST /api/coupons/redeem`, `GET|POST /api/coupons/me`.
- **Admin:** `/coupons` lists all codes + all redemptions and creates fully-configurable **admin** codes. Amounts (`REFERRAL_RECIPIENT_EUR`=10, `REFERRAL_REFERRER_EUR`=5, `REFERRAL_WINDOW_DAYS`=14) are `app_setting`s editable in admin `/settings` (only affect newly-activated referral codes).
- **Referral ↔ subscription NOW wired (both-sided, subscriber-only):** the Stripe webhook's `checkout.session.completed` (subscription) handler calls `settleReferralRewards(subscriberId)` after its tx commits (it opens its own tx). `settleReferralRewards` now grants **both** the redeemer's deferred welcome bonus AND the referrer reward (past-deadline → both `expired`), idempotently. **Creation gate is built:** `POST /api/coupons/me` returns `403 {error:"subscription_required"}` unless `userHasActiveSubscription(userId)` (`web/lib/stripe/subscriptions.ts`, `subscription.status='active'`) — the `CouponTab` shows the "Abo only" hint instead of the activate button. **Our coupons ≠ Stripe's built-in coupons/promotion codes** (which are payment discounts on a Payment Link — orthogonal, not used). Existing activated codes keep working if a sub later lapses (only creation is gated). New `coupon_redemption` cols: `recipient_reward_status` (default `granted` = backfill-safe) + `recipient_reward_granted_at`.

## Stripe billing (Payment Links) — Abo pro App + Guthaben-Top-Ups
Monetisation via **static Stripe Payment Links** (created in the Stripe dashboard, pasted into admin `/settings`), parameterised at click time — NOT dynamically-created Checkout Sessions. Two products: a **5€/month per-app hosting subscription** (each active sub unlocks that app's future premium features + grants its owner **5€/month expiring builder credit**) and one-time **top-ups (5/10/20€, persistent credit)**.
- **No serving gate.** The free `<slug>.apps.<APPS_DOMAIN>` subdomain always stays online; `readPublishedFile()` is untouched. A subscription only flips `project.hostingActive` (denormalised entitlement for FUTURE features: custom domain, more storage/traffic) — nothing in the serve path reads it yet.
- **Attribution:** `buildCheckoutLinks`/`withCheckoutParams` (`lib/stripe/links.ts`) append `?client_reference_id=…&prefilled_email=…` to the base links. Subscription link → `client_reference_id = projectId`; top-ups → `= userId`. The webhook resolves the **paying user by the pinned email** and only sets `user.stripeCustomerId` on them; it links the app **only if `project.userId === payingUserId`** (ownership check — a crafted link can't attach a sub to someone else's project or hijack the portal).
- **Webhook** `app/api/stripe/webhook/route.ts` (public, `runtime=nodejs`): reads the RAW body (`request.text()`), verifies with `stripe.webhooks.constructEventAsync(raw, sig, STRIPE_WEBHOOK_SECRET)`. `handleStripeEvent` (`lib/stripe/webhook.ts`) processes each event in **one tx** that also inserts a `stripe_event(id)` dedupe row (`ON CONFLICT DO NOTHING` → re-delivery skips; a throw rolls back the whole tx so Stripe's retry re-runs cleanly). Handlers are upsert-first/order-independent. Events: `checkout.session.completed` (sub → subscription row + hostingActive + settleReferralRewards; payment → top-up by `amount_total/100`, EUR-guarded), `invoice.paid` (grant 5€ monthly `credit_grant`, `expiresAt = invoice.lines[0].period.end`, idempotent by unique `stripe_invoice_id`; only `billing_reason` create/cycle), `customer.subscription.updated|deleted` (sync status; `hostingActive = status===active`), `invoice.payment_failed` (past_due). Field accessors are **version-defensive** (Stripe moves `invoice.subscription`, `current_period_end`) — the client pins no `apiVersion`.
- **Two credit sources:** persistent `user_credit.balanceEur` (free grant + coupons + top-ups, no expiry) + a new **`credit_grant` ledger** (expiring monthly sub credit, one row per invoice, own `expiresAt`, stacks across subs). **Spendable = balanceEur + Σ(remainingEur WHERE expires_at>now())** (lazy expiry, no cron). `recordUsageAndDeduct` (`lib/cortecs/billing.ts`) draws from the **soonest-expiring grant first** (SELECT … FOR UPDATE — mirrors `redeemCoupon`; the bare balance UPDATE alone doesn't cover the separate grant rows under READ COMMITTED), remainder from `balanceEur`. `getBalance`/`ensureCredit`/`getBalanceBreakdown` all sum both; `/api/credit` returns the split (`persistentEur`/`monthlyEur`/`monthlyExpiresAt`).
- **Other tables/columns:** `subscription` (per-app, keyed by unique `stripe_subscription_id`), `user.stripe_customer_id` (portal). All new → `uniqueIndex` not `.unique()`, migrate-safe.
- **UI:** account modal 4th tab "Abo & Guthaben" (`SubscriptionTab` in `user-menu.tsx`) — credit split, top-up buttons, per-app host/manage — backed by `GET /api/billing/overview`; `POST /api/stripe/portal` opens the billing portal (gated on `stripeCustomerId`). Returning from a link (`/app?checkout=success`) `workspace.tsx` polls `/api/credit` (webhook is async) + fires `CREDIT_CHANGED_EVENT`. **No SSE** — the only stream is request-scoped to an agent run.
- **Env:** `STRIPE_SECRET_KEY` + `STRIPE_WEBHOOK_SECRET` are **secrets (env-only**, read in `lib/stripe/env.ts`); the Payment Link URLs + `SUBSCRIPTION_MONTHLY_CREDIT_EUR` are non-secret `app_setting`s (config accessors in `lib/cortecs/config.ts`, admin "stripe" settings group). All wired in the `web` compose `environment:` block + root `.env.example`. **Setup:** create the links + a webhook endpoint (`…/api/stripe/webhook`) in the Stripe dashboard, set the link success_url to `…/app?checkout=success`, paste the signing secret → `STRIPE_WEBHOOK_SECRET`, set `STRIPE_SECRET_KEY` (restricted key) → redeploy. Unset keys → webhook 503s + buttons hidden (fail-safe).
